Spring Boot Migration – Technical Architecture Evolution (WebSphere to OpenShift)

1. Executive Summary

The current application landscape is based on a vendor-provided SWIFT processing system deployed on WebSphere 8.x running on RHEL, which has now reached an end-of-life state. Over time, the platform has accumulated architectural constraints, particularly around filesystem-based integration, tight coupling with application server capabilities, and limited scalability.

To address these limitations and align with enterprise platform strategy, the application is being transitioned to a Spring Boot–based architecture deployed on OpenShift. This transition is not just a technology upgrade, but a fundamental shift in integration, deployment, and operational model.

The immediate objective is to deliver v17 on Spring Boot within the current year, maintaining functional parity, while enabling a platform foundation for further evolution in v18 and beyond.

---

2. Current Architecture (WebSphere-Based Model)

The current system is deployed as a WAR application on WebSphere Application Server 8.x, hosted on a RHEL-based infrastructure.

2.1 Application Runtime Model

The application relies heavily on WebSphere-provided services:

- MQ connectivity is handled through the WebSphere MQ JMS provider and resource adapter
- Application lifecycle, threading, and connection pooling are managed by WebSphere
- Scheduling is implemented via cron jobs on the RHEL server

This results in a model where the application is tightly coupled to the application server runtime, with limited portability.

---

2.2 Integration Model (Filesystem-Based SWIFT Processing)

A key characteristic of the current system is its filesystem-driven integration model:

- Inbound SWIFT files are placed on a shared NAS location
- The application reads files from an input directory
- Upon processing:
  - Successfully processed files are moved to a processed folder
  - Failed files are moved to an error/failure folder
- Outbound SWIFT files are generated by the application and written back to the same NAS location for downstream consumption

This approach introduces:

- strong dependency on shared storage
- challenges in scalability and concurrency
- tight coupling between systems through file movement rather than message-driven flow

---

2.3 Messaging and Data Layer

- IBM MQ (version 9.3) is used in the environment, though the application interacts via WebSphere-managed JMS abstraction
- Oracle 19c is used for persistence
- MQ client version is embedded within WebSphere (resource adapter), not explicitly managed by the application

---

2.4 Scheduling Model

Batch processing is controlled through:

- cron-based scheduling on RHEL servers
- direct invocation of application processes or scripts

This model:

- lacks central orchestration
- is not aligned with enterprise scheduling standards
- becomes difficult to manage in distributed/containerized environments

---

2.5 Key Limitations of Current State

The current architecture presents several constraints:

- End-of-life platform (WebSphere 8.x and RHEL)
- Tight coupling to application server runtime
- Filesystem-based integration limiting scalability
- Lack of cloud/container readiness
- Distributed scheduling with limited control
- Hidden dependency on MQ client via WebSphere

These limitations drive the need for modernization.

---

3. Target Architecture (Spring Boot on OpenShift)

The target state introduces a modern, containerized, and message-driven architecture, aligned with enterprise platform standards.

---

3.1 Application Runtime Model

The application is deployed as a Spring Boot service running in OpenShift containers.

Key characteristics:

- Runs as stateless application pods
- Scales horizontally across multiple instances
- No dependency on application server (WebSphere removed)
- Runtime managed by OpenShift platform

This introduces:

- improved scalability
- platform portability
- standardized deployment model

---

3.2 Integration Model (MQ-Based Processing)

The most significant architectural shift is the move from filesystem to MQ-based integration.

Inbound Flow:

- Upstream systems publish messages to an MQ input queue
- Application consumes messages directly from the queue

Processing:

- Business logic is executed within the Spring Boot service

Outbound Flow:

- Processed messages are published to an MQ output queue
- Downstream systems consume messages from MQ

Failure Handling:

- Failed messages are routed to an error queue / dead letter queue (DLQ)

This replaces:

- input folders → input queues
- processed folders → message consumption / logging
- error folders → error/DLQ queues

---

3.3 Data Layer

- Oracle 19c remains unchanged
- Application connects directly using standard database drivers
- No dependency on application server data source abstraction

---

3.4 Scheduling Model

In the target state:

- Scheduling is handled via Spring Boot scheduler (application-level scheduling)

This is a shift from OS-level cron jobs to:

- application-controlled scheduling
- centralized within the application runtime

(Subject to future alignment with enterprise scheduler if required)

---

3.5 Platform Capabilities (OpenShift)

The OpenShift platform provides:

- Container orchestration
- Auto-scaling across pods
- Load balancing
- Readiness and liveness probes for health monitoring
- Centralized logging and monitoring integration

This removes dependency on:

- physical servers
- static infrastructure
- manual failover mechanisms

---

4. Key Architectural Evolution (Current vs Target)

Runtime

- WebSphere-managed → Spring Boot standalone runtime
- Server-based → Container-based (OpenShift)

Integration

- Filesystem/NAS → MQ-based messaging
- Batch file processing → Event/message-driven processing

MQ Usage

- WebSphere-managed JMS → Explicit MQ client integration

Scheduling

- Cron jobs → Application-level scheduler

Scalability

- Limited vertical scaling → Horizontal scaling via pods

Deployment

- Manual/server-based → CI/CD-driven container deployment

---

5. Strategic Benefits

The target architecture enables:

- Standardization on enterprise OpenShift platform
- Improved scalability and resilience
- Decoupled integration via MQ
- Removal of legacy filesystem dependencies
- Better alignment with cloud-native principles
- Simplified operations and monitoring

---

6. Migration Approach

Phase 1 – v17 (Current Scope)

- Migrate application to Spring Boot
- Deploy on OpenShift
- Maintain functional parity
- Shift integration to MQ-based model
- Replace cron with application scheduler

Phase 2 – v18 (Future)

- Further architectural optimizations
- Platform enhancements
- Potential alignment with enterprise scheduler and observability stack

---

7. Conclusion

This migration represents a controlled but significant architectural transition, moving from a monolithic, server-dependent model to a modern, containerized, and message-driven architecture.

While v17 focuses on enabling this transition with minimal disruption, it establishes a foundation for future scalability, resilience, and enterprise alignment.
